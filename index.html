Accessible LLM Interaction for Blind and Screen-Reader Users: Biases & Improvements

Keywords: Accessibility; Blind users; Screen reader; Bias & fairness; HCI evaluation; LLM;

This project investigates how blind and screen-reader users interact with LLMs, asking: what interaction breakdowns and biases arise, and what design or alignment changes would improve the experience? To enable rigorous study, we built a screen-reader-first chat system (Next.js + Edge streaming) that both supports accessible interaction (reliable announcements, predictable focus, keyboard workflows) and collects structured research data—user intents, complete transcripts, and sentence-level judgments with consent. The instrumented platform allows fine-grained analysis of dialog quality and bias patterns in non-visual use, informing concrete UI fixes and accessibility-focused alignment signals.

★Features:

  • Accessible interaction model: ARIA live regions with deterministic focus across streaming & dialogs; full keyboard workflows (Enter/Space actions, Cmd/Ctrl+Enter newline, Meta/Ctrl+Alt+0 to open annotation); “Annotate last response” quick action with safe targets.
  • Sentence-level bias/quality annotation: Segments assistant output into sentences (with Intl.Segmenter fallback) and renders each as a focusable button; captures qualitative judgments, improvement suggestions, and consent; persists rich metadata (chatId, messageIndex, sentenceIndex, length, wordCount) in MongoDB.
  • Standardized research data collection: Optional pre-chat intent capture; automatic end-chat feedback after five exchanges; complete timestamped transcripts with SSR-safe serialization to support qualitative coding and quantitative analysis.
  • Streaming & context control: Real-time OpenAI Edge streaming; persisted assistant responses post-stream; optional history summarization endpoint to manage token limits during longer sessions.
  • Privacy & integrity: Auth0 authentication/middleware, per-request session checks, SSR owner verification with ObjectId validation; explicit consent flag (shareResponse); all writes scoped to the authenticated user.
  • Study operations: Protected chat list for participants, intent-aware “New chat” flow, input locking during streaming/dialogs to keep the screen-reader narrative stable during data collection.

Skills: Next.js (React), Node.js, MongoDB, Auth0, Tailwind CSS; Web Accessibility (WAI-ARIA, focus management, SR workflows); OpenAI Edge streaming; REST APIs; research instrumentation & data modeling.

Virtual Offshore Wind Turbines: Unity-Based VR Simulation for Mooring Line Design Evaluation

Keywords: VR visualization; Unity; Offshore wind; Mooring design; Stakeholder engagement; Underwater visualization

Floating offshore wind (OSW) is planned for the Gulf of Maine, but fishing communities need clear, spatially accurate ways to understand turbine arrays and mooring layouts above and below the waterline. Working in the Insite Lab at Colby College, I built an immersive Unity/VR experience that lets stakeholders explore turbine spacing, seabed habitats, and multiple mooring-line configurations. The simulation combines guided tours, free exploration, and interactive UI so experts can evaluate compatibility with existing fishing gear and practices.

★Features:

  • Developed advanced VR locomotion: a guided boat tour along a Bezier path + comfort-safe teleportation for free exploration.
  • Engineered water physics, lighting, and post-processing for realistic surface/underwater scenes; improved measured user engagement by ~35%.
  • Built interactive virtual controls (buttons/panels) with Unity XR Interaction Toolkit to toggle mooring designs, sea states (calm/active/storm), and habitat overlays.
  • Modeled underwater environments (rocky seabed, kelp/coral, sand) and integrated physics/materials for believable visuals.
  • Maintained & upgraded the project to the latest Unity LTS for better performance and device compatibility (HTC Vive Pro + controllers).
  • Presented the work at the 2024 Colby Undergraduate Summer Research Retreat (CUSSR); awarded 2nd place.

★Skills: Unity (C#), XR Interaction Toolkit, SteamVR, physics & lighting, Shader Graph/post-processing, scene optimization, Blender asset workflow, usability/engagement evaluation, Git.

Netify — Playlist Migration from NetEase to Spotify

Timeline: May 2025 · Stack: Next.js 14 · TypeScript · Tailwind CSS · FastAPI (Python) · Spotify Web API · NetEase API · Vercel · Fly.io

Netify addresses music lock-in by reliably transferring entire NetEase Cloud Music playlists to Spotify. The app authenticates securely via OAuth 2.0 PKCE, fetches large playlists with pagination and fallbacks (handling the NetEase ~800-track limit), and uses a fuzzy matching pipeline (duration tolerance + artist normalization) to find the best Spotify tracks. A progress UI and a missing-tracks report make long transfers transparent. Since launch, Netify has helped 50+ users migrate 10,000+ songs.

  • Secure auth & session: Spotify OAuth 2.0 with PKCE; callback, token exchange, and session state managed in React Context.
  • Large-playlist handling: Robust pagination/chunking and graceful fallbacks for NetEase API limits (~800 tracks/playlist).
  • Smart matching: RapidFuzz with duration tolerance and artist normalization cut mismatches by ≈50% vs title-only.
  • Full transfer: Creates Spotify playlists with tracks, name, and cover; progress bar + detailed missing-track report.
  • Deployable & fast: Next.js frontend on Vercel; FastAPI backend on Fly.io; bilingual UI (简体中文 / English).

Skills: Next.js (React), TypeScript, Tailwind CSS, FastAPI (Python), OAuth 2.0 PKCE, REST APIs, RapidFuzz, pagination & resilience patterns, cloud deployment (Vercel/Fly.io).

Light — RPG Game Prototype

Timeline: Jan 2023 · Team: 5 · Engine: Unreal Engine 5

Light is a vertical-slice RPG prototype exploring AI-driven enemies and light-based puzzle mechanics. In Agile sprints, our team built a playable loop that combines patrol/chase enemies with laser–mirror–sensor puzzles to test how systemic AI pressure and spatial reasoning affect player engagement. Internal playtests showed a ~50% increase in engagement after iterating on puzzle clarity and combat pacing.

  • Team leadership & process: Led a 5-person team through Agile sprints (backlog, stand-ups, playtest-driven iteration); produced a 14-minute demo to showcase the slice.
  • AI enemies: Implemented patrol, random roam, and player-chase behaviors with state logic and pathfinding; tuned perception and chase windows for readable difficulty.
  • Puzzle design: Built interactive laser emitters, mirrors, and sensors to gate progression; iterated on level hints and fail-safe resets to reduce player confusion.
  • Game feel & audio: Composed original soundtrack/SFX in GarageBand and integrated cues to heighten combat intensity and puzzle feedback.
  • Playtesting & QA: Ran multi-round tests to find bugs, triaged issues, and measured engagement time and completion rates to guide tuning.

Skills: Unreal Engine 5, gameplay programming & AI logic, level/puzzle design, playtesting & telemetry, Agile collaboration, audio production (GarageBand), demo production.
